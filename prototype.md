# bad-roads — прототип анализа жалоб на дороги

## Краткое описание

Интерактивное демо для клиента **«Томская региональная управляющая компания по ЖКХ»**: пользователь загружает (или делится) потоком русскоязычных текстов про дорожные проблемы Томска. По умолчанию браузер подтягивает **живые заметки из OpenStreetMap API** для bbox города через внутренний маршрут `GET /api/tomsk-notes` и смешивает их с синтетическими строками (явная маркировка источника) до целевого объёма. Сервер для свободного текста извлекает локации эвристикой или языковой моделью, оценивает тяжесть (1–10), нормализует адрес для геокодинга и отображает скопления на интерактивной карте (**Leaflet** + кластеры; подложка **Esri World Light Gray** с резервом **CARTO** для устойчивости CDN на Vercel). Дополнительный модуль принимает фотографию и запускает сегментацию по англоязычному текстовому классу.

## Предметная область

Городские дороги, эксплуатация улично‑дорожной сети, обратная связь жителей (неформализованный текст + опционально фото).

## Тип задачи

`language` + визуальная локализация (опционально): **NER / извлечение локаций**, **оценка тональности/тяжести**, **геокодирование**, **картографическая агрегация**; отдельный поток — **сегментация изображения**.

## Входы

- Сырой многострочный русский текст жалоб и/или машиночитаемые строки с координатами заметок OSM (`[[osm-note …]]`).
- Автоматически: текст из `GET /api/tomsk-notes` (верхнее поле на главной после монтирования).
- (Опционально) изображение участка дороги + текстовый класс на английском для модели сегментации.

## Выходы

- JSON со списком точек: исходное описание / адрес или идентификатор заметки OSM, нормализованная подпись, широта/долгота (геокодер или координаты заметки), оценка 1–10, краткое резюме, частота упоминаний, уверенность, сервисные предупреждения.
- Для сегментации: нормализованный JSON с превью/оверлеем (data URL), списком полигонов/bbox, диагностикой (`processingMode`, `upstreamStatus`, …).

## Подход

1. **Текст**: вызов настроенного языкового бэкенда (`vllm` или `anthropic` в порядке `DEMO_FOUNDATION_PROVIDERS`) только для «обычных» строк (комментарии `#` удаляются, строки `[[osm-note …]]` отделены во встроенную ветку без геокодирования координат). При ошибке — детерминированные эвристики по шаблонам улиц. Для строк OSM считываются точные координаты из разметки.
2. **Датасет первого экрана**: HTTP к OSM `/api/0.6/notes` bbox Томска, парсинг XML на сервере; при нехватке строк — добавление синтетики с указанием, что строки воспроизводят общие тематики дорожных статей региональных медиа, а не текст заметки OSM.
3. **Геоданные**: серверные запросы к **Photon**, при неудаче — к **Nominatim**; контекст «Томск, Россия». Для встроенных OSM строк геокодирование координат не требуется.
4. **Карта**: **Leaflet** + **MarkerCluster**; базовые тайлы **Esri** (World Light Gray) с однократным переключением на **CARTO** после серии ошибок загрузки; радиус маркера масштабируется по частоте.
5. **Сегментация**: прокси к `${SAM3_API_BASE_URL}/sam3-image` с `x-api-key`, уменьшение изображений >1024 px через `sharp`, нормализация ответов и локальный визуальный fallback.
6. **Anthropic без отдельного прокси**: если `ANTHROPIC_PROXY_URL` / `ANTHROPIC_BASE_URL` отсутствуют и задан `ANTHROPIC_API_KEY`, вызов выполняется к `https://api.anthropic.com/v1/messages`.

## Ограничения и риски

- Нет размеченного датасета; качество извлечения определяется промптом и конкретностью текста.
- Геокодирование может давать пропуски или смещения; заметки OSM тематически не фильтруются к «дорогам» автоматически.
- Сегментация чувствительна к ракурсу, освещению и формулировке `text_prompt`.
- Прототип **не** подтверждает фактическое состояние покрытия в полевых условиях.
